# 诊断与改进说明（0720）

> 这是一次性的**复盘/交付说明**，不是技能运行所需组件（技能跑起来不依赖本文件）。
> 记录：本次 agent 运行暴露了什么、哪些是 skills 问题、哪些是 agent 问题、改了什么、以及遗留风险。

## 一、到底是什么情况

`post-evaluation`（后评估小助手）在**第4步生成文件**时崩掉，报 JSON 解析错误：

```
json.decoder.JSONDecodeError: Expecting ',' delimiter: line 4 column 146 (char 268)
```

agent 随后陷入**反复 `replace` / `sed` 打地鼠**：改一处引号→重新解析→又在别处报错→再改……越改越多，始终修不完。

### 根因链（谁的问题，分清）

**skills 侧（换任何合理 agent 都会踩）：**
- **S1**〔高〕第4步**强制模型手工合并、手打整份含引号中文 JSON**。而此时磁盘上 `table2.json`/`table3.json` 已是上游脚本 `json.dump` 出的合法 JSON——技能把机器已做对的事又交回模型手打，把安全性丢了。**这是破裂的结构性根因。**
- **S2**〔高〕`generate_tables.py` 用**裸 `json.load`**，失败只抛**字节偏移**（`char 268`），无行号/成因。换谁都定位不到是哪个引号，只能盲试。**打地鼠是这个不可读报错逼出来的。**
- **S3**〔低·残留〕即便用合并脚本，第2步仍需**就地编辑 `table2.json`** 填三列（佐证材料要引用报告"存在的问题"原文，易带半角引号）。手写面从"整份"缩到"三列"，未完全消除。

**agent 侧（换个更强的 agent 就能避免；skills 改不动它的手）：**
- **A1**〔高〕**诊断对了却用错药**：已判断"引号未转义是遍布全文的系统性问题"，却用针对特定子串的 `.replace()` 去修——全局问题用局部补丁，数学上不可能收敛。
- **A2**〔高〕**源头就不该手打**：手上已有两份合法 JSON，正确做法是 `load` 两份→合并 dict→`dump`，全程库负责转义；它却手抄正文把引号打成半角。
- **A3**〔中〕**失败后不止损、不换路**：第一次扩大 replace 没收敛就该停，它反而又试 `sed` 配 `“`（sed 不解析 unicode，可预判失效）、再回去 replace。

> 共因：A1/A2 主责在 agent 的行为，但 S1+S2 是**放大器**——S1 把它推进注定失败的手写路径，S2 又让它拿不到能自救的报错。**skills 能做的，就是把这两个放大器拆掉。**

### 一个重要的设计取舍：不做静默"自动修复"

早期版本让加载器在解析失败时**自动转义未转义引号并继续**。复盘后**撤掉了主路径上的这个行为**，原因：

- 转义判据是**启发式**（"半角 `"` 后紧跟 `,:}]` 才算字符串结束符"），**并不健全**：当正文引号恰好紧跟半角标点（如 `他说"是的",我同意`、`比例"3":1`）会**误判**、截断或改坏文本；
- 本技能明确要求"**照抄原文一字不改**"、产出供专家**法律评审**。一个可能**悄悄改动 verbatim 正文**的修复，比直接失败更危险。

因此改为：**主路径严格解析 + 响亮失败 + 处方式报错**；自动转义仅保留在**显式 `--fix`** 命令里（会改写文本、且提示必须人工核对），不在生成流程里默默发生。

## 三、验证（在本仓库沙箱实跑，非空口）

- 用**基础版**脚本喂含未转义半角引号的 `combined_data.json` → 复现 `Expecting ',' delimiter (char 268)`。
- 改后：坏 JSON → `generate_tables.py` 给出定位到行、带一次性做法的报错并退出（**不再静默改文本**）。
- `merge_tables.py` 合并两份合法分表 → `generate_tables.py` 正常生成 4 个 docx/xlsx。
- 合法 JSON（含全角/转义引号）→ 原样解析、不改动。
- 已知局限：`--fix` 的启发式在"正文引号紧跟半角标点"时会误判（实测会导致修复后仍解析失败，从而响亮报错而非静默产出错文档）——所以它只作显式、需人工核对的兜底，不进主路径。

## 四、遗留与建议

- **S3 未完全闭合**：第2步仍要就地编辑 `table2.json`。缓解靠"就地小改不重排 + 全角引号 + 处方式报错"，但引用带引号原文时风险仍在。若要彻底消除，可再往前做一步：让填三列也走结构化输入（如单独的填空脚本/表单）而非手改 JSON。
