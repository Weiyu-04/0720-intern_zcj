# 诊断与改进说明（0720imporved 分支）

> 基础版本见 `main` 分支；本分支（`0720imporved`）在其上做修复。

## 一、到底是什么情况

`post-evaluation`（后评估小助手）在**第4步生成文件**时崩掉，报的是 JSON 解析错误：

```
json.decoder.JSONDecodeError: Expecting ',' delimiter: line 4 column 146 (char 268)
```

agent 随后陷入**反复 `replace` 打地鼠**：改一处引号→重新解析→又在别处报错→再改……越改越多，始终修不完。

### 根因（一句话）

**这不是脚本的算法 bug，而是"让模型手写整份 JSON"这一步太脆。**

工作流第4步要求模型把 `table2` 和 `table3_groups`**手工合并成一份 `combined_data.json`**。事故调查报告正文里**大量出现引号**，例如：

> （参见报告**"三、（三）存在的问题**·1.山东中诚公司**"**）

模型在手写/重排这段中文时，很容易把本该是全角 `"` `"` 的引号打成**半角 ASCII 双引号 `"`**。而 JSON 用半角 `"` 作字符串定界符——**字符串值内部出现未转义的半角 `"`，解析器就以为字符串在这里提前结束了**，于是抛 `Expecting ',' delimiter`。报告里这种引号有很多处，所以"改一处又冒一处"。

### 为什么原脚本会把小问题放大成灾难

`generate_tables.py` 里是**裸 `json.load(f)`**，没有任何容错或提示：

```python
with open(inp, encoding="utf-8") as f:
    data = json.load(f)          # 一旦失败只抛一个字节偏移量
```

报错只给 `char 268` 这种**字节偏移**，模型看不出"是哪种错、该怎么改对"，只能盲目试探——于是打地鼠。

### 澄清：其它 `json.load` 不受影响

`production-safety-accident-review` 里也有几处 `json.load`，但读的都是**随包发布的静态数据**（`geo.json`、`gbt4754.json` 等，机器生成、天然合法），**不是模型手写的**，不受此问题影响。整个仓库里唯一读"模型手写 JSON"的点，就是 `generate_tables.py`。**问题面很小、很精准。**

上游两个确定性脚本 `segment_table2.py`、`parse_table3.py` 用的是 `json.dump(..., ensure_ascii=False)`，**会正确转义引号**，产出的 `table2.json`/`table3.json` 本身是合法的。**坏 JSON 是在"手工合并"那一步才被引入的**——第4步等于把机器已经做对的事又交回给模型去手打，把安全性丢掉了。

## 二、改了什么（本分支）

目标：**一处杂散引号再也不该让整条流水线翻车**。三项改动，只动 `post-evaluation`：

| 文件 | 改动 | 作用 |
|---|---|---|
| `scripts/robust_json.py` | **新增** | 容错 JSON 加载器：①先严格解析（合法则原样不动）；②失败则自动修复字符串内"非结构性"的未转义半角引号后重试；③仍失败则打印**定位到行、带 `^` 指示符**的可读报错，并点明"半角→全角引号"这一最常见原因。 |
| `scripts/merge_tables.py` | **新增** | 安全合并器：读第2、3步的 `table2.json` + `table3.json`，用 `json.dump` 写出 `combined_data.json`——**从源头保证合并结果永远合法**，第4步不必再手打整份 JSON。 |
| `scripts/generate_tables.py` | 改 1 处 | 把裸 `json.load` 换成 `load_json_tolerant`。**其它逻辑一字未动。** |
| `SKILL.md` | 改第4步 | 指引优先用 `merge_tables.py` 合并；正文引号一律用全角 `"` `"`；说明加载器会自动兜底、报错已可读，不要再逐个 replace。 |

### 修复的判定逻辑（保证不误伤合法 JSON）

逐字符状态机，只在"字符串内部"动手：一个半角 `"` 只有当其后第一个非空白字符是 `, : } ]` 或文件结束时，才算真正的字符串结束符；否则判定为正文引号并转义。已经是 `\"` 的转义引号会跳过、不二次转义。**严格解析成功时完全不触发修复**，因此合法文件绝不被改动。

## 三、验证（均已实测通过）

1. **复现**：用原始官方脚本喂含未转义半角引号的 `combined_data.json` → 复现 `Expecting ',' delimiter (char 268)`。
2. **自动修复**：改进版 `generate_tables.py` 喂同一份坏文件 → stderr 提示"已自动修复 4 处"，正常生成 4 个 docx/xlsx，且正文引号原样保留在文档里。
3. **合并路径**：`merge_tables.py` 合并两份分表 → `generate_tables.py` 生成 4 个文件，成功。
4. **不误伤**：含全角引号与转义引号的合法 JSON → 原样解析，不改动。
5. **真语法错误**：缺逗号的结构性错误 → 给出定位到行、带 `^` 的可读报错并退出（不乱修）。

## 四、给使用方的一句话建议

集成时把 `post-evaluation/scripts/` 整个目录一起带上（新增了 `robust_json.py`、`merge_tables.py`）。运行时 Python 需装 `python-docx`、`openpyxl`、`PyMuPDF`（见 `README-集成说明.md`）。
