# 生产安全事故研判助手

## 用途

给应急指挥中心处理"警情信息 / 事故快报 / 热线翻单"类文本，对**一起**事件产出三项标准结论，并附完整依据链：

1. **事故分类**（事故类型，GB 6441-2025 的 27 类基本事故类型）
2. **事故等级**（一般 / 较大 / 重大 / 特别重大）
3. **事故所属行业领域**（管理分类 + 国民经济行业分类 + 按行业的 WSA 编码）

一切结论必须**有迹可循、有理可依**：每一步都要引用到具体条款，数值和编码不允许模型臆断。

> **定位与免责**：本工具提供的是警情阶段的**辅助初判**（此阶段信息常不完整），作用是统一判定口径、留存依据链、明确标注待核实项；**事故的性质、等级、分类、所属行业领域最终以依法开展的事故调查结论为准**。研判结果不构成最终认定，需经调查核定，相应责任不由工具的确定性表述承担。

## 路径规则

- 将本轮实际加载的 `SKILL.md` 所在目录记为 `SKILL_DIR`（例如 `/任意位置/production-safety-accident-review`）。
- 本文档内所有 `references/…`、`scripts/…`、`templates/…`、`examples/…` 均相对 `SKILL_DIR`；调用脚本时用真实目录，如 `python3 "<SKILL_DIR>/scripts/grade.py" --deaths 1`，不要把尖括号原样传给 shell。
- 不要假设 Skill 固定在 `/mnt`、用户目录或某个绝对路径，也不要用 `find`/`which` 去找脚本；脚本内部已用相对自身位置定位数据文件，按真实路径调用即可跨环境运行。

## 判定总流程（严格按顺序，逐步为闸门）

按下列 1→5 顺序执行。任一前置步骤为"否/不纳入"时，其后步骤停止判定但要说明理由。

| 步 | 判什么 | 依据文件 | 方式 |
|---|---|---|---|
| 1 | 是否**生产安全事故** | `references/01-是否生产安全事故.md` | 语义判断 |
| 2 | 是否**纳入统计** | `references/02-是否纳入统计.md` | 规则匹配 |
| 3 | **事故等级** | `references/03-事故等级-国务院令493号.md` + `scripts/grade.py` | 脚本精确计算 |
| 4 | **事故类型（27类）** | `references/04-事故类型-GB6441-2025.md` | 对照表1 |
| 5 | **行业领域 + WSA 编码** | `references/05-行业领域对照表.md` + `scripts/industry_signal.py` + `scripts/encode.py` | 闸门 → 语义判 → 查表编码 |

> 建设状态：五步链条与确定性件均已建成——要素抽取(`references/00`)、是否事故(`01`)、纳统(`02`)、等级(`03` + `grade.py`)、事故类型(`04`)、行业(`05` + `industry_signal.py` 闸门 + `encode.py` + 全1381小类码库)，含两则端到端样例(`examples/样例.md`：一则信息充足能定案、一则信息真空只能列候选)。地名确定性归一(`scripts/geo.py`，离线，全国省/地/县)已建成——**凡有地点需归一即用它**（见下"要素抽取"：地点字段交 `geo.py`，模型不自行判定），不是每案强制（无地点/地点"未掌握"时不调）。

## 第一步之前：要素抽取

抽取的标准产物是一张固定字段、**带原文出处**的**结构化抽取卡**（不是自由行文），见 `references/00-要素抽取与语义理解.md`——先出卡、再按 1→5 判定；数值/编码/地点字段分别交给 `grade.py`/`encode.py`/`geo.py`，模型不自行计算。护栏：拿不准一律"未掌握/待核实"，绝不臆造。重点：数字常不明说需语义推断；死亡/重伤/**直接经济损失**三维并重；"未掌握"与"已知为0"必须区分；喂 grade 前须扣除军警消救援伤亡、并入急性工业中毒为重伤。要点如下：

先从警情文本中抽取以下要素（缺失的标注"未掌握"，不臆造）：

- **涉事单位及性质**：名称；是否生产经营单位（企业/事业单位/个体工商户/其他组织）。
- **是否生产经营活动**：含预备性、收尾性活动；施工/安装/运输/开采等。
- **活动/作业类型**：用于判定事故类型与行业领域。
- **伤亡与损失**：死亡（含下落不明）人数、重伤（含急性工业中毒）人数、直接经济损失（万元）。
- **人员身份**：是否本单位人员、单位外居民/行人、抢险救援人员（军警消不计入等级统计）。
- **车辆类型**（如涉车）：是否属十二类道路运输车辆。
- **责任与单位归属**：交叉作业、劳务分包、租赁、外包工程等归属线索。
- **可能的核销排除项**：人为故意（恐怖/投毒/纵火/盗窃等，经公安认定）、突发疾病、不可抗自然灾害。

抽取后，按 1→5 逐步判定。

## 事故等级（第三步）：调用脚本，禁止心算

```
python3 "<SKILL_DIR>/scripts/grade.py" --deaths <死亡> --serious <重伤> --loss <直接经济损失万元>
```

直接引用脚本返回的 `basis` 作为等级依据。术语口径见 `references/03-事故等级-国务院令493号.md`。

## 行业领域（第五步）：先过闸门，禁止凭印象补主营

**每案必调，不得跳过**（同 grade.py 的口径）。行业判定最常见的失效不是判错，而是**在输入根本没给行业信息时，凭对某家公司的既有印象补出一个"主营"，然后一路推到唯一小类**——此时模型自认"有信息"，`references/05` §29/§30 那些以"无信号"为触发条件的规则**不会被触发**。故用确定性闸门先把这一步锁住：

```
python3 "<SKILL_DIR>/scripts/industry_signal.py" scan   --text "<警情全文>" --name "<单位全称>"
python3 "<SKILL_DIR>/scripts/industry_signal.py" verify --claim "<你拟填入『单位主营』的内容>" --text "<警情全文>" --name "<单位全称>"
```

- `scan` 判输入有没有行业线索（词表有局限，只作提示）；
- **`verify` 是判准**：它检查你拟填的『单位主营』能否在**警情原文或单位名称**里逐字找到。
  **verify 说"无出处" → 『单位主营』只能填"未掌握"，且禁止给出唯一四位小类与确定的 WSA 编码**，
  改按 `references/05` §29 列候选、注明"待定到唯一行业后出码"、提示需补充单位主营确认。
- 闸门**只阻止答案、不提供答案**：是哪个行业，仍由你据 `references/05` 判。
- ⚠️ 主营"未掌握" **≠** 行业"无法判断"——候选照列（§30 只在连候选都列不出时才用）。
- 来源限定的完整口径见 `references/00` 的「单位主营行业」字段说明。

## 输出

严格套用 `templates/输出模板.md`：纯文本、书面语、固定六段、**无 emoji 与图标**，每段给"结论—要素—依据"。

**输出洁癖（面向用户的研判结果）——总原则：研判要写得像一位应急专家手写的，不像程序日志。**
- **只给"结论 + 法规/标准依据"，绝不描述任何内部计算过程。** 一律不得出现：脚本名（grade.py/encode.py/geo.py/classify_org.py/industry_signal.py）、内部文件/路径（references/…、scripts/…、SKILL_DIR）、以及任何暴露机制的字眼——"脚本""查表""编码脚本""计算件""启发式归类""经……确认""我调用了/我读取了/查表得到"等，全部不进最终结果。
- **可以引用的**：法规/国标条款（如"《生产安全事故报告和调查处理条例》第三条第（四）项""一般规则第6条""GB 6441-2025 第5.2、5.5条"）——这些是正当依据，照写。**区别在于**：写"依据 GB 6441-2025 第5.5条，其编码为 WSA35431"是对的；写"经编码脚本查表确认……WSA35431"是错的（暴露了机制）。
- **举一反三的反/正例**：
  - 反例：`据 classify_org.py 启发式归类，名字无法判定` ／ `经编码脚本查表确认小类5431，编码 WSA35431` ／ `调用地名归一件得出上海市长宁区`
  - 正例：`单位名称无行业指示，结合事件线索暂无法定到唯一行业，列候选如上，需补充单位主营确认` ／ `国民经济行业为普通货物道路运输（小类5431），按行业分类的事故编码 WSA35431（依据 GB 6441-2025 第5.2、5.5条）` ／ `事故发生地为上海市长宁区`
- 不写"以上分析""综上""核心要点如下"等套话；不复述本说明的存在。
- 未掌握/待核实项如实标注，但用面向用户的自然语言，不用内部术语。

## 当前覆盖范围与口径

- **范围**：行业对照表（`references/05`）与编码（`scripts/encode.py` + 全 1381 小类码库）**覆盖全部六个管理分类**（A / B / C·F·H / E / G / D·I—T），任何行业的警情都能完整落域并编码。B 采矿业、C·F·H 商贸制造业是当前工作重点，但不是能力边界。
- **事故类型口径**：主输出以 GB 6441-2025 的 27 类为准，并附 1986 版 20 类映射（供直报系统填报），见 `references/04`。
- **行业编码精度**：WSA3XXXX 后四位为 GB/T 4754-2017 四位小类代码（去门类字母，如 内河货物运输 5523 → WSA35523）。已获《2017 国民经济行业分类注释》完整代码表（1971 条代码），可精确到四位小类，由 `scripts/encode.py` 查表校验后输出，位数不允许模型臆断。
- **行业判定的信号与缺口**：确定行业需"事故发生单位的主营行业"。主判据是**事故中的作业/主营活动描述**（语义，模型据 `references/05` 判，防"活动≠主营"背离）；`scripts/classify_org.py` 据公司名给"登记类型 + 行业候选"作离线交叉核对（启发式、仅参考，名字无信号如"某某实业"则待核实）。二者都定不了时标"待核实单位主营"，不臆断。
  **"答案的缺席"已经确定性化了**：`scripts/industry_signal.py` 判"本次输入有没有行业信号"、`verify` 判"主营这一栏抄不抄得出出处"——这两件事是**输入文本的性质**，不是世界的性质，零企业名录即可判定，故已落为代码闸门（见上"行业领域（第五步）"）。**待数据的增强**（把"答案"本身也确定性化）：权威企业名录（信用代码→行业码，离线查表）——数据到位即可；在线查询因与"离线可移植"约束冲突且涉隐私，默认不做。**在名录到位之前，闸门保证的是"没有出处就不许给准数"，而不是"给不出准数就不干活"——候选照列。**

## 依据文件清单

- `references/` 五步判定依据（结构化后的法规/制度条文）。
- **运行时脚本**：`scripts/grade.py` 等级（**每案必调**）、`scripts/industry_signal.py` 行业信号闸门（**每案必调**，`scan` 提示 + `verify` 判主营有无出处）、`scripts/encode.py` 行业→WSA 编码、`scripts/geo.py` 地名归一（省/地/县）、`scripts/classify_org.py` 公司名→登记类型+行业候选（启发式，仅作交叉核对）；数据在 `scripts/data/`（`gbt4754.json`/`geo.json`）。
- **维护脚本**（研判**不调用**，仅在重建/更新码库后运行）：`scripts/maintenance/build_catalog.py`、`build_geo.py`（从 Excel 重建码库）、`validate.py`（校验码库与源 Excel 全量逐条一致）。
- `templates/输出模板.md` 标准输出格式。
- `examples/样例.md` 两则端到端已标注样例：**案例一**信息充足→定到唯一小类出码；**案例二**信息真空→主营填"未掌握"、只列候选、码待定。两则合起来才是完整口径，别只学案例一。
