# 标准输出格式

要求：纯文本、书面语、结构固定；不使用 emoji、图标、装饰性符号；每一步结论都附具体条款依据，做到有迹可循、有理可依。研判按下列六段呈现，逐段"结论—要素—依据"。**写得像应急专家手写，不像程序日志**：只给结论 + 法规/国标依据，**不描述任何内部计算过程**——脚本名、"脚本/查表/编码脚本/计算件/启发式/经…确认/我调用了"等机制字眼、内部文件路径，一律不进最终结果；也不写"综上/以上分析/核心要点"套话。（正例：`…事故编码 WSA35431（依据 GB 6441-2025 第5.5条）`；反例：`经编码脚本查表确认…WSA35431`）

若某一步判定为"否"或"不纳入"，其后步骤标注"不再判定"，但仍需说明理由。

前置（内部流程，不体现在输出里）：六段研判由结构化抽取卡产出——先出卡（各字段带原文出处、未掌握项如实标注），数值/编码/地点分别由内部工具确定性得出，再组织成下列六段；工具与过程一律不写进最终结果。

## 输出骨架

事故研判结果

一、是否生产安全事故
    结论：是 / 否
    关键要素：涉事单位及性质、是否生产经营活动、有无人身伤亡或直接经济损失
    依据：《生产安全事故统计调查制度》调查对象和统计范围 +（如触及核销情形）相应排除条款

二、是否纳入统计
    结论：纳入 / 不纳入 / 先行填报待调查认定
    关键要素：命中的具体情形（如十二类道路运输车辆、单位外人员受害、100万元门槛等）
    依据：《生产安全事故统计调查制度》"事故统计一般规则"第 X 条

三、事故等级
    结论：一般 / 较大 / 重大 / 特别重大
    关键要素：死亡 X 人、重伤 X 人、直接经济损失 X 万元
    依据：《生产安全事故报告和调查处理条例》（国务院令第493号）第三条第（X）项，及命中区间
    （本项等级由等级计算件确定，直接引用其依据句；结果中不出现脚本名）

四、事故类型
    结论：<GB 6441-2025 表1 的类型名称>（类型编码 WSA1XXXX）
    关键要素：事故直接原因、经过特征
    依据：GB 6441-2025 表1 第 X 项及其说明

五、所属行业领域
    （能定到唯一行业时）
    管理分类：<如 C·F·H 商贸制造业 — 工贸>
    国民经济行业分类（GB/T 4754-2017）：<门类 / 大类 / 中类 / 小类>
    按行业分类的事故编码：WSA3XXXX
    依据：《生产安全事故统计调查制度》附录（二）"管理分类与国民经济行业分类对照表"对应项

    （信号指向多个可能行业时，列出候选而非"待核实"）
    可能行业（按可能性排序，需补充单位主营确认）：
      候选1：<行业/管理分类>（若为…）— <WSA3XXXX 或 待定>
      候选2：<行业/管理分类>（若为…）— <WSA3XXXX 或 待定>
      …
    编码：待补充单位主营、定到唯一行业后据以生成
    依据：事件仅见<通用设备/线索>，据此列出上述可能行业；具体需营业执照经营范围/实际产品或作业确认

六、研判说明
    简述关键要素抽取过程、存疑点、需进一步核实的信息，以及超出当前覆盖范围时的说明。
    **单位主营未掌握时，须在此外显一句自然语言的来源交代**（抽取卡是内部产物、不出厂，
    若此处不写，读者无从知道"主营"那一栏是有出处的还是补出来的）。
    写法示例：「单位主营未掌握：警情未述及产品、工艺或场所类型，单位名称亦无行业指示，故未填。
    行业据事故经过的通用线索列候选如上，需补充单位主营（营业执照经营范围／实际产品或作业）后
    定到唯一小类并出码。」——自然语言，不提任何工具、文件或机制。

## 完整样例

两则端到端的完整研判样例见 `examples/样例.md`，作为本模板的 few-shot 参照与回归基准。为避免重复维护导致口径漂移，本模板不再单独保留一份样例，统一以 `examples/样例.md` 为准。

**两则必须一起看，缺一不可**：
- **案例一（信息充足）**：定到唯一小类、出 WSA 码——示范上面第五段的**前半个**骨架；
- **案例二（信息真空）**：单位主营无出处→填"未掌握"、只列候选、码待定——示范第五段的**后半个**骨架（"可能行业…"）与第六段的来源交代。

只学案例一会得到一个"永远给准数"的研判器：输入没给行业信息时，它会凭印象补一个主营再推出唯一小类。案例二就是用来堵这个的。
