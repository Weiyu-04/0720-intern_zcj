# 表3 分组顺序修改说明（手动修改指引）

**对象技能**：`post-evaluation`（后评估小助手）
**主改文件**：`post-evaluation/scripts/parse_table3.py`
**同步文件**：`README.md`
**性质**：代码分组发射顺序与规范文件不一致，属代码疏漏；本说明给出逐处的手动修改方法。

---

## 一、修改目的

使表3（事故处理落实情况评估表）的分组**输出顺序**符合规范 `post-evaluation/references/表3规则表.md` 第二节：

> 组1 行刑衔接情况 → 组2 行政处罚情况 → **组3 企业内部处理情况** → **（有则加）其他处理情况**

即：**企业内部处理情况固定为第 3 组；其他处理情况"有则加"，出现时排在企业内部处理情况之后。**

---

## 二、修改前的问题

`parse_table3.py` 的 `parse()` 函数用四个 `if` 代码块依次拼装各组，每个块调用 `hdr()` 分配组号：

```python
def hdr(title):
    nonlocal gi
    gi += 1                       # 组号按 hdr 被调用的先后递增
    rows.append({"no": str(gi), "person": title, ...})
    return gi
```

**组号由 `if` 块在源码中的先后顺序决定。** 修改前四个块的顺序是：

```
if criminal(刑事) → if admin(行政处罚) → if other(其他处理) → if disc(企业内部处理)
```

因此当一份报告**同时含"其他处理"和"企业内部处理"两组**时：
- 其他处理先被 `hdr()` 分到组 **3**；
- 企业内部处理后分到组 **4**。

结果与规范相反（应为 内部=组3、其他=组4）。子行序号 `组号.序号`（如 3.1、4.1）也随之错位。

> 注：本问题只影响**组的排位**，不影响各组内容抽取（分类、责任人、处理建议均正确）。仅当两组同时出现时才显现。

## 四、逐处修改（照抄即可）

### 修改点 1（核心）：交换 `if disc:` 与 `if other:` 两块的顺序

在 `parse_table3.py` 的 `parse()` 函数中，找到下面这段（`if other:` 在前、`if disc:` 在后）：

**【当前代码】**
```python
    if other:
        g = hdr("其他处理情况")
        for k, o in enumerate(other, 1):
            rows.append({"no": f"{g}.{k}", "person": o["unit"],
                         "suggestion": f"建议{o['dept']}依法依规给予处理。",
                         "provider": o["dept"], "materials": MAT_OTHER, "note": ""})
    if disc:
        g = hdr("企业内部处理情况")
        for k, d in enumerate(disc, 1):
            rows.append({"no": f"{g}.{k}", "person": d["disp"], "suggestion": d["sug"],
                         "provider": discipline_provider(d["affil"], d["item"]),
                         "materials": MAT_DISC, "note": ""})
```

**【改为】**（把 `if disc:` 块整体提到 `if other:` 块之前，并加一行说明注释）
```python
    # 依据 references/表3规则表.md 第二节：企业内部处理情况固定为第3组，
    # 其他处理情况"有则加"排在其后。hdr() 按调用先后分配组号，故必须先写
    # if disc（企业内部），再写 if other（其他）。两块顺序不可调换。
    if disc:
        g = hdr("企业内部处理情况")
        for k, d in enumerate(disc, 1):
            rows.append({"no": f"{g}.{k}", "person": d["disp"], "suggestion": d["sug"],
                         "provider": discipline_provider(d["affil"], d["item"]),
                         "materials": MAT_DISC, "note": ""})
    if other:
        g = hdr("其他处理情况")
        for k, o in enumerate(other, 1):
            rows.append({"no": f"{g}.{k}", "person": o["unit"],
                         "suggestion": f"建议{o['dept']}依法依规给予处理。",
                         "provider": o["dept"], "materials": MAT_OTHER, "note": ""})
```

> 说明：只调换两个 `if` 块的先后位置，块内代码一字不改。调换后 `hdr()` 先给企业内部处理分组号（3）、再给其他处理（4）。

### 修改点 2：更新文件开头 docstring 的顺序说明

### 修改点 5（建议）：新增回归测试，防止顺序被再次改回

在 `parse_table3.py` 末尾，`if __name__ == "__main__":` 这一行**之前**，插入下面的函数：

```python
def _selftest():
    """回归测试：企业内部处理情况必须排在其他处理情况之前
    （依据 references/表3规则表.md 第二节 与客户口径）。
    构造同时含"其他主管部门处理"与"企业内部处分"两组的最小虚构报告
    （全部为占位名称，不含任何真实案例内容），断言两组的相对顺序与组号。"""
    rpt = (
        "四、对有关责任人员及单位的处理建议\n"
        "（一）建议给予行政处罚的单位\n"
        "1. 某某有限公司，建议某某县应急管理局对某某有限公司依法予以行政处罚。\n"
        "（二）建议行业主管部门处理的单位\n"
        "建议某某县市场监督管理局对某某租赁有限公司依法依规给予处理。\n"
        "（三）建议企业内部处分的人员\n"
        "1. 某某，某某有限公司安全员，建议由某某有限公司给予其记过处分。\n"
        "五、事故整改和防范措施\n"
    )
    order = [(r["no"], r["person"]) for r in parse(rpt)["table3_groups"][0]["rows"]
             if r["person"] in GROUP_TITLES]
    names = [p for _, p in order]
    assert "企业内部处理情况" in names and "其他处理情况" in names, ("两组都应出现", order)
    assert names.index("企业内部处理情况") < names.index("其他处理情况"), \
        ("企业内部处理必须排在其他处理之前", order)
    numof = {p: int(n) for n, p in order}
    assert numof["企业内部处理情况"] < numof["其他处理情况"], ("组号应内部<其他", order)
    print("selftest OK")
```

然后把文件末尾的 `if __name__ == "__main__":` 块开头：

**【当前代码】**
```python
if __name__ == "__main__":
    data = parse(open(sys.argv[1], encoding="utf-8").read())
```

**【改为】**（增加 `--selftest` 入口）
```python
if __name__ == "__main__":
    if len(sys.argv) > 1 and sys.argv[1] == "--selftest":
        _selftest()
        sys.exit(0)
    data = parse(open(sys.argv[1], encoding="utf-8").read())
```

> 说明：`sys` 已在文件顶部 `import re, sys, json` 中导入，无需另加导入。

## 六、判定依据与待确认事项

**依据**：
1. 规范文件 `post-evaluation/references/表3规则表.md` 第二节「三级分组与序号（务必遵守）」明确写明：企业内部处理情况 = 组序 3、其他处理情况 = （有则加）。

**待确认（重要）**：上述客户样板**只有 3 个分组、不含"其他处理"组**，因此它未直接演示"其他处理排在企业内部处理之后"这一点。本次修改依据的是**规范文件的明文**与**客户口径**。**建议在改动落地前，向客户再确认一次"其他处理是否排在企业内部处理之后"**；如客户口径不同，只需将修改点 1 的两个 `if` 块顺序再次对调，并相应更新修改点 2/3/4 与回归测试即可。
