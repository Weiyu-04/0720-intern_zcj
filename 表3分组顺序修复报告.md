## 0. 摘要

表3（事故处理落实情况评估表）按"处理类型"分组。规范要求：**企业内部处理情况固定为第 3 组，其他处理情况"有则加"排在其后**。但 `parse_table3.py` 的代码发射顺序是 `刑事 → 行政处罚 → 其他处理 → 企业内部处理`，导致**当报告同时出现"其他主管部门处理"和"企业内部处分"两组时，企业内部处理被挤到第 4 组、其他处理占了第 3 组**，与规范相反。本次通过**交换两个代码块的顺序**修复，并补齐文档与回归测试。

| 项 | 内容 |

| 现象 | 两组同现时，组顺序为 …→其他处理→企业内部处理（其他在前） |

| 严重度 | 正确性问题（建议在"速度重构"之前先修，避免重构叠加更难定位） |

### 关于客户样板的诚实说明（重要）

客户样板**只有 3 个分组、全表不含任何"其他"单元格**，因此它**从未把"其他"与"内部"放在一起排过**，无法**直接**证明"其他必须排在内部之后"这条。它能坐实的只是"企业内部处理 = 组 3（紧跟行政处罚）"。

因此本次修复的**决定性依据是**：① 规范文件 `表3规则表.md` §二「务必遵守」的明文；② 用户（对接客户方）转述的客户口径。客户样板为**旁证**、非该具体排序点的铁证。

> **建议的收尾动作**：向客户再确认一句"其他处理是否排在企业内部处理之后"，或取得一份**同时含两组**的真实表以终极坐实。目前代码/测试/文档已按此口径钉死；若客户口径不同，回改也只是再换一次两块顺序 + 更新回归测试。

## 3. 根因分析

`parse_table3.py` 的 `parse()` 用**四个 `if` 块**依次拼装各组，每块调 `hdr()` 发一个组号：

```python
def hdr(title):
    nonlocal gi
    gi += 1                       # 组号 = "第几个被发射的组"
    rows.append({"no": str(gi), "person": title, ...})
    return gi
```

**组号完全由 `if` 块在源码里出现的先后决定**。修复前顺序为 `criminal → admin → other → disc`，于是两组同现时 `other` 先拿到较小的号（3）、`disc` 拿 4 → 内部被挤到第 4 组。子行号 `f"{g}.{k}"` 里的 `g` 跟着错。

**这是纯发射顺序问题，不是分类/抽取逻辑问题。**

## 4. 完整改动清单

共 4 处对齐 + 1 处新增测试；权威规范文件不动。

| # | 位置 | 改动 | 性质 |
|---|---|---|---|
| 1 | `parse_table3.py` `parse()` 内 | **把 `if disc:`（企业内部）整块移到 `if other:`（其他）之前** | 核心，决定组号 |
| 2 | `parse_table3.py` docstring | 顺序声明改成 内部=3、其他=有则加 | 文档一致 |
| 3 | `parse_table3.py` `GROUP_TITLES` 元组 | 顺序调成 内部在其他前 | 一致性（见 §5） |
| 4 | `README.md` §5 表格 | 企业内部处理调到其他处理前 | 交接表一致 |
| 5 | `parse_table3.py` `_selftest()` | **新增 `--selftest` 回归测试** | 防回退 |
| — | `references/表3规则表.md` | **不动** | 本就正确，是代码违反的 spec |

### 核心改动（真实 diff，逻辑一字未改，仅换位 + 加锁定注释）

```diff
+    # 客户口径 / references/表3规则表.md §二「务必遵守」：企业内部处理固定为第3组，
+    # 其他处理情况"有则加"排在其后。故必须先发 disc(内部)、再发 other(其他)——hdr() 按
+    # 发射先后给组号，两块顺序即决定组号(内部=3、其他=4)。切勿调换。
     if disc:                       # ← 原来在 other 之后，现移到前面
         g = hdr("企业内部处理情况")
         for k, d in enumerate(disc, 1):
             rows.append({"no": f"{g}.{k}", "person": d["disp"], ...})
     if other:                      # ← 原来在 disc 之前
         g = hdr("其他处理情况")
         for k, o in enumerate(other, 1):
             rows.append({"no": f"{g}.{k}", "person": o["unit"], ...})
```

## 5. 边界考量（工程完备性检查）

- **子行序号**：`f"{g}.{k}"` 的 `g` 自动跟随组号变对（内部 3.x、其他 4.x），无需额外处理。
- **`GROUP_TITLES`（改动 3）**：它仅用于 `if r["person"] in GROUP_TITLES` 判"是否组标题行"，属**成员判断、与顺序无关**——功能上**不改也不影响**；改它纯为让"声明顺序"和"实际输出"一致。
- **`generate_tables.py`**：按 `rows` 既有顺序渲染、无硬编码组序 → **自动跟随**，无需改。
- **无"其他"组的常见情况**：只有内部时，`admin → disc` 使内部仍是紧跟行政的那组（客户样板正是刑事1/行政2/内部3）→ **行为不变**。
- **同一人受多种处理不去重**：分处各组、与组序无关 → 不受影响。
