## 0. 目标

在公司 Agent 平台「中城交智枢」里做一个 **后评估小助手** 技能：用户上传《事故调查报告》，依据《生产安全事故后评估指南》自动生成两张评估表，每张各出 **Word + Excel**（共 **4 个文件**），并展示智能体思考过程。

- 《**事故整改和防范措施落实情况评估表**》（指南里编号表2）
- 《**事故处理落实情况评估表**》（指南里编号表3）

不做《自评估报告评估表》（表1，人工补）；不做评估报告（后续阶段）。

> 下文沿用"表2/表3"这个内部简称，但**技能对用户不依赖这四个字**——用户说"做个后评估""生成那两个表"甚至说错编号，都能正常工作（见 SKILL.md 开头"怎么理解用户的提问"）。

1. **答案不泄漏**：`5·6`、`12·22` 两案例的**已填好评估表 = 测试集**。技能里的知识只能来自 附录A定义 + 指南通用示例 + 通用领域知识，**绝不能把这两个案例的答案写进技能**（SKILL.md / references / 模板都不行）。客户给的样板 xlsx 同理——**只可用于对齐格式与规则，不可抄其评估内容答案**。

## 5. 怎么起服务

**坑**：项目路径含中文，Python 的 `.pth` 在非 UTF-8 环境解码中文路径失败 → 找不到 `zcjagent`；且 `uv run` 每次卸载重装本地包，并发会崩。**所以不要用 `uv run`**（跑测试脚本也一样，直接用 `.venv/bin/python`）。

```bash
# 后端（或双击根目录 启动.command）
cd MainAgent/backend
set -a && source ../.env && set +a
LANG=en_US.UTF-8 LC_ALL=en_US.UTF-8 PYTHONIOENCODING=utf-8 PYTHONUTF8=1 PYTHONPATH=".:packages/harness" \
  .venv/bin/python -m uvicorn app.gateway.app:app --host 0.0.0.0 --port 8001 --reload

# 前端
cd MainAgent/frontend && pnpm dev

# OnlyOffice 容器（双击根目录 启动OnlyOffice.command）
```

- 健康检查：`curl http://localhost:8001/health`；前端 3000；OnlyOffice 8080
- 若报 `No module named zcjagent`：`cd backend && uv pip install -e ./packages/harness`
- 停服务：`停止.command`（**不含容器**；容器用 `docker stop onlyoffice-ds`，设了 `--restart unless-stopped`，Docker 一开就自己回来，不必重拉镜像）
- 首次需在 `/setup` 建管理员账号（密码用户自设，助手不代填）
- 后端日志有 `No 'checkpointer' section … InMemoryStore … Thread list will be lost on server restart` 警告：基座未配 `checkpointer` 段，**LangGraph 的 store 用了内存实现**。不过实测 sqlite（`backend/.zcj-agent/data/zcjagent.db`）里 `threads_meta`、`checkpoints` 都有数据，对话内容与生成的文件都不丢。**"列表是否真会在重启后消失"未实测确认**——若真遇到，加 `checkpointer: {type: sqlite, connection_string: ...}` 即可（结构见 `config/checkpointer_config.py`）。

**环境要点**：默认模型 `deepseek-v4-flash` @ `https://llm.zcjsh.com/v1`（`$DEEPSEEK_API_KEY` 在 `.env`），`temperature: 0.2`；`sandbox.allow_host_bash: true`（技能要跑 python）；技能跑的 `python` = `/opt/miniconda3/bin/python`，已装 python-docx / openpyxl / PyMuPDF。

**摘要配置（2026-07-17 调过，重要）**：`config.yaml` 的 `summarization` 段
| 项 | 原值 | 新值 | 原因 |
|---|---|---|---|
| `trigger` tokens | 32000 | **80000** | 本技能系统提示词（SKILL.md + 4 参考）实测占 **25.5k token 且每轮重发**，按 32000 触发时只剩 ~6.5k 给对话 → **两三个来回就摘要** |
| `keep` messages | 10 | **40** | 摘要后多留原文，上一轮的 warnings/表格数据/用户要求不易被压掉 |

- **为什么必须改**：摘要中间件用 `RemoveMessage(REMOVE_ALL_MESSAGES)` **从 state 真删消息**，而**界面渲染的就是 state 的 messages** → 用户会眼睁睁看着历史对话消失。这不是"模型看不到"而已。
- `summarization.*` 是 per-run 字段，**改完下一条消息即生效，不用重启**。
- ⚠️ **未解风险**：查不到 `deepseek-v4-flash` 的上下文窗口（config 没配、网关 `/v1/models` 不返回元数据）。**若模型上下文 < 80k，会在摘要触发前就报 `context length exceeded`**。真遇到 → 改回 32000，或改用 `type: fraction, value: 0.8`（按模型上限比例触发，不依赖具体数字、换模型也安全，config 里有注释掉的现成选项）。
- ⚠️ **另一隐患**：本技能消息里有大块头（脚本输出 JSON、报告正文、表格数据），**40 条本身可能达数万 token**，可能出现"摘要完还是超"。若如此，改 `keep` 为 `type: tokens, value: 8000` 更可控。
- **更根本的优化方向**（未做）：25.5k 系统提示词每轮重发是真正的浪费，其中判别特征库 42 条占大头、却只在"匹配评估内容"那步用得上。改成**按需 read_file 读 references**，每轮固定开销可从 25.5k 降到几 k，阈值也就不必调这么高。改动不小，供权衡。

## 6. 技能结构与架构

目录：`MainAgent/backend/skills/public/post-evaluation/`
菜单挂接：`backend/skills/domains.json` → `emergency-dispatch`(应急指挥调度) 下槽位「后评估小助手」→ `skill_name: post-evaluation`

### 核心架构：代码管结构，LLM 管语义

> 拆行/定主体/表3解析这类**句法结构**任务用**确定性代码**（模型做这个会 9/13/6/13 行乱跳）；模型只做**真语义**（评估内容匹配 42 条特征库）。
> **代码为主 + LLM 强制兜底校验**：代码遇没见过的格式会失手，所以它**自报 `_diagnostics.warnings`**，由 LLM 对照原文抓漏补错。**兜底是"敢用代码"的前提，也是防过拟合的解药。**

```
第1步 extract_clean      剥页码/脚注 → 干净正文
第2步 segment_table2     表2骨架(序号/拆子行/主体) → LLM 校验 warnings + 填评估内容/评估方式
第3步 parse_table3       表3三级分组全解析          → LLM 校验 warnings
第4步 generate_tables    出 4 个文件 → present_files 全部交付 + 思考过程
```

### 文件清单

| 文件 | 作用 |
|---|---|
| `SKILL.md` | 人设 + 上述工作流 + **"怎么理解用户的提问"**（不依赖"表2/表3"字样）+ **列归属**（序号/整改要求归脚本，模型一字不许改）|
| `references/表2表3填写规范.md` | 两表逐列总规范 |
| `references/评估内容判别特征库.md` | **42 条**（= 附录A 全部条目），每条含 定义/判别特征/区分边界/通用示例。表2「评估内容」匹配的核心教材 |
| `references/附录A评估方式速查.md` | 编号 → 评估方式（含合并单元格继承：2.6.2←2.6.1、5.1.2-5.1.5←5.1.1、5.2.2←5.2.1）|
| `references/表3规则表.md` | 表3 分组 + 4 类模板 |
| `scripts/extract_clean.py` | 版面感知提取：按字号（**自适应阈值**）剥页码/脚注/内联标记，被脚注劈开的句子自动拼回 |
| `scripts/segment_table2.py` | **表2 确定性拆行/定主体** |
| `scripts/parse_table3.py` | **表3 确定性解析**（分类→4模板→三级分组→编号）|
| `scripts/generate_tables.py` | 产 4 文件（docx 用模板克隆行；xlsx 用 openpyxl 现绘，同源 JSON）|
| `templates/表2模板.docx`、`表3模板.docx` | 官方格式模板（`.bak` 是改动前的原件）|

### 泛化要点（**改代码前必读，勿只按一份报告调**）

- **小标题有三种排版**：A型独占一行无句号（5·6）/ B型带句号且正文挤同行（12·22）/ 跨行断在词中（12·22 "…安全管控措"+"施。"）。已用「物理行为主 + 句号为辅 + 判断句号前是否要求句」兼容三种。**只按句号切会把 5·6 的首条要求吞进标题（丢内容）**。
- **拆子行只认顿号「、」，不认「及」**："属地政府**及**市安委会有关成员单位"是一个整体主体。
- **主体识别** = 具名主体 + 模态词（要/应/应当/须）；无模态词时用动词兜底（加强/落实/督促…）并标**中置信度**交 LLM 复核。
- **泛称展开**（"各相关单位"→具体单位）与**名称规范化**（"市交通行政管理部门"→市交通委）由 LLM 做，代码只负责标出来。

## 7. 客户口径（**最高优先依据**）

样板文件：**`生成评估表/表2和表3-【1222事故】-增加地标对应列.xlsx`**（客户 2026-07-16 提供）
> 优先级高于旧范例 docx 与旧模板。sheet「表2」「表3」（Sheet3 是空表）。
> "**地标对应列**" = 评估内容 / 评估方式，即对应地方标准《后评估指南》**附录A** 的条目。
> ⚠️ 它是 12·22 的**填好答案**（测试集）——只可对齐格式规则，**不可抄答案**。

### 表2（8 列）
`序号 | 整改要求 | 整改主体 | 评估内容 | 评估方式 | 佐证材料 | 整改情况 | 说明`

- **章节标题 (一)(二)(三) 单独成行**：序号=1/2/3，整改要求=小标题，**其余列填 `-`**
- **数据行** `N.M`；多个具名单位并列 → 拆子行 `N.M-1 / N.M-2`，第2行起整改要求写「同上」
- **整改要求**：**逐字照录、脚本产出，模型一字不许改**（曾出现模型"顺手精简"删掉"塔吊平衡重"那句的严重错误）
- **评估内容**：填「**编号 + 附录A 条目原文**」（不是简称）；**客户明确：匹配度 30-50% 即可**，靠语料库后续提升，不必过度纠结
- **评估方式**：查速查表、合并去重（不单独判断）
- **佐证材料**：客户注明「**先空着，后续细化开发**」→ **本期非验收重点**
- **整改情况**：留空，三档「符合／基本符合／不符合」

### 表3（7 列 + 三级分组）
`序号 | 事故责任人员/单位（括号内为隶属单位）| 处理建议 | 提供材料单位 | 佐证材料 | 处理情况 | 说明`

分组 = 组标题行（序号=N，第2列=组名，其余列留空）+ 子行 `N.1/N.2`：

| 组 | 内容 | 提供材料单位 |
|---|---|---|
| **行刑衔接情况** | ③刑事，**全部人员合并为一行** | **办案的公安、检察院、法院** |
| **行政处罚情况** | ①行政处罚，逐个一行 | **被处罚方自己**（个人取其所属单位）—— **不是市应急局**！市应急局只是作出处罚的机关 |
| （**其他处理情况**） | ④其他主管部门（有则出现） | 相应主管部门 |
| **企业内部处理情况** | ②内部处分，逐个一行 | 责任人所属单位（可写"A 或其上级 B"）|

- 「四、(一)不予追究责任」的人员（如已死亡）**不列入表3**
- **同一人受多种处理 → 各组各列一次，不去重**
- **处理情况**：留空，**三档**（符合/基本符合/不符合）

## 9. 踩过的坑（省时间用）

| 坑 | 真相 |
|---|---|
| **OnlyOffice 报"您正在尝试执行您没有权限的操作"** | **和权限、JWT 都无关**。真凶是**文档 key 太长**：中文文件名经 json 转义每字 6 字符，key 破 300，而 OnlyOffice 规定 ≤128、DB 是 varchar(255)。**基座原有 bug**（任何中文长文件名都会炸），已修成 32 字符摘要 |
| **OnlyOffice 排查心法** | **它的前端报错基本不可信，一律先 `docker logs onlyoffice-ds`**，真话都在那儿 |
| **File not found in thread outputs** | `thread_dir(user_id=None)` 会**静默退回 legacy 布局**（缺 `users/{uid}/` 段）。必须 `uid = user_id or get_effective_user_id()` |
| **容器"假就绪"** | 容器内 nginx 先于内部服务起来，healthcheck 会先 `true` 再变 502。要**连续 3 次成功**才算真就绪 |
| **模型把原文"顺手精简"了** | 已在 SKILL 钉死列归属：序号/整改要求归脚本，模型只能改 主体/评估内容/评估方式/佐证材料 |
| **max_tokens 给小了** | reasoning 吃光额度 → JSON 截断 → 正文为空。给 ≥32000 |
| **项目路径含中文** | 别用 `uv run`（`.pth` 解码中文路径失败）。用 `.venv/bin/python` + 显式 PYTHONPATH + UTF-8 环境变量 |
| **改了技能却没生效** | 旧对话用的是旧系统提示，**必须新建对话** |
| **网页上历史对话消失了** | 摘要中间件 `RemoveMessage` **从 state 真删消息**，界面读的就是 state。根因是系统提示词占 25.5k 且每轮重发，按原阈值 32000 只剩 6.5k 给对话 → 两三轮就摘要。已上调至 80000 / keep 40（见 §5）。**别误判成"数据丢了"——sqlite 里都在** |
